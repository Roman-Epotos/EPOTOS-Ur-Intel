2. ТЕХНОЛОГИЧЕСКИЙ СТЕК
=============================================================
Next.js 14, TypeScript, Tailwind CSS, Supabase (cloud) + pgvector
AI основная: OpenRouter (google/gemini-2.5-flash)
AI PDF сравнение: OpenRouter (anthropic/claude-sonnet-4-5) — ПЛАТНАЯ! только ai-document-compare

Storage: Supabase (buckets: contracts PUBLIC, templates PUBLIC, counterparty-docs PUBLIC)
Хостинг: Vercel Pro (оплачен и подключен с 03.07.2026)
Генерация .docx: JSZip (прямая замена {{field}} в XML)
PDF извлечение текста: unpdf — нестабилен на Vercel! Показываем ошибку
Связанный модуль: ЭПОТОС-Core (https://epotos-core.vercel.app) — отдельный
  деплой, хостит RAG-ассистента (/api/assistant/chat, /api/assistant/search)
  на pgvector, используется ЮрИнтел через простой fetch (без общего кода)

10. АРХИТЕКТУРНЫЕ РЕШЕНИЯ
=============================================================
Безопасность:
  auth_id → sessionStorage → /api/bitrix/verify → Б24 profile API
  Без sessionStorage И без URL params → редирект на Б24 (только если self===top)
  window.self === window.top → прямая ссылка без авторизации
  Мобильный Б24: нет auth_id → разрешаем (storedUser без auth_id)

Тихое обновление токена Б24 (/api/bitrix/refresh):
  POST { refresh_id } → grant_type=refresh_token к oauth.bitrix24.tech/
    oauth/token/ (client_id/secret из ENV) → возвращает новую пару
    { auth_id, refresh_id, member_id, domain }
  useBitrixAuth.ts: в sessionStorage-ветке, если verifyData.valid === false
    И storedUser.refresh_id существует → вызываем /api/bitrix/refresh,
    при успехе — обновляем sessionStorage и setUser БЕЗ reload/редиректа;
    при неудаче — старое поведение (sessionStorage.removeItem + reload)
  refresh_id уже сохранялся в sessionStorage и раньше (спред ...data.user
    из /api/bitrix/callback), просто не использовался — теперь используется
  ОГРАНИЧЕНИЕ: работает только для сценария «страница уже открыта, токен
    протух со временем» (отдельная вкладка/ПКМ). НЕ решает прямой клик
    внутри iframe чата Б24 — там contract_id теряется ДО того, как вопрос
    валидности токена вообще встаёт (Б24 сам режет query-параметры при
    встраивании iframe, это происходит раньше нашего кода)

ЭДО (упрощённый алгоритм):
  Финансист → ГД → решение (уведомление в чат Б24) → способ подписания
  Крон /api/cron/edo-reminder (07:00) — напоминание через 3 дня

Напоминания о сроках согласования:
  Крон /api/cron/approval-reminder (06:00 UTC = 09:00 МСК)
  3 типа: approval_deadline_soon/reached/overdue

Отмена отклонения согласования:
  GET /api/approvals: .in('status', ['active','completed','cancelled'])
  revoke/route.ts: wasRejected → реактивация сессии+контракта
  Фронт: hasSession включает 'cancelled'; isSessionActive гейтит кнопки

Роль «Наблюдатель»:
  role='observer' → status='observing' сразу; исключён из allApproved
    (проверки === 'required'); реактивация сессии пропускается для observer

Лист согласования:
  HTML-строка → Blob → window.open — не jsPDF (кириллица)
  Верхняя toolbar с window.print(), скрывается в @media print

Учёт оплат / дебиторка (contract_payments):

    paid_total, remaining), POST (canRecordPayment проверка), DELETE

11. ПРАВИЛА РАБОТЫ
=============================================================
1. Новые файлы — New-Item в PowerShell (относительный путь, если терминал
   уже в папке проекта — не указывать полный путь заново, риск опечатки)
2. Редактирование — ТОЛЬКО Ctrl+H в VS Code
3. Не более 3 блоков кода за раз, ждать подтверждения
4. Перед деплоем КОДА — npm run build
5. CONTEXT — без build, сразу git push
6. При зависании деплоя — git commit --allow-empty + git push
7. Repomix:
   npx repomix --config repomix-api.config.json
   npx repomix --config repomix-components.config.json
   npx repomix --config repomix-pages.config.json
8. Деплой (три команды построчно в одном блоке):
   git add -A
   git commit -m "..."
   git push
9. Если фрагмент не найден → обновить repomix, ИСКАТЬ ТОЧНЫЙ путь файла
   через grep/awk по repomix перед предложением замены — никогда не
   угадывать путь файла или содержимое кода. Один и тот же import/паттерн
   (например useBitrixAuth) может встречаться в десятках файлов — для
   уникальности str_replace включать соседние уникальные строки, а не
   искать одну строку саму по себе
10. git log завис → нажать q
11. check_xml.py: INPUT = 'ИмяФайла.docx'
12. Диск проекта — E:\ (см. раздел 1); при работе с git revert/checkout
    учитывать, что repomix-файлы тоже версионируются и могут дать конфликт
    при merge/revert — разрешать через git checkout --theirs на них
13. Разные страницы одного модуля могут быть server-компонентами (без
    хуков, например app/contracts/[id]/page.tsx) — хуки/context добавлять
    в дочерний client-компонент (например ContractTabs.tsx), а не в саму
    server-страницу

12. ПРЕДЛОЖЕНИЯ ИЗ ВНЕШНИХ AI-КОНСУЛЬТАЦИЙ (Qwen/Gemini/ChatGPT/DeepSeek)
=============================================================
Проект показывали на ревью четырём внешним ИИ (Qwen3.7-Plus, Gemini, ChatGPT,
DeepSeek) для независимой оценки. Общая оценка — зрелый CLM-продукт
корпоративного уровня. Из конкретных, проверенных на реальном коде находок
(не общих слов) отобрано следующее — в порядке приоритета:

1. ⬜ getCompanyPrefix.ts — устранить дублирование парсинга company_prefix.
   Паттерн startsWith('Э-К') ? 'Э-К' : split('-')[0] уже дважды ловил баг
   (signed-documents, вероятно есть и другие места). Создать
   app/lib/getCompanyPrefix.ts, использовать везде вместо инлайн-парсинга.

2. ✅ ВЫПОЛНЕНО 06.07.2026 — ручной ввод факта оплаты в карточку договора
   (contract_payments, см. разделы 4/6/10)

3. ⬜ Хэш документа в листе согласования — для юридической значимости/
   compliance (доказательство неизменности файла с момента согласования).
   Добавить вычисление хэша (SHA-256) подписанного файла, отображать в
   листе согласования рядом с примечанием.

4. ⬜ Перенос SPECIAL_SIGNERS/ADMIN_IDS из хардкода в Supabase-таблицы
   (system_settings/company_signers) — сейчас смена сотрудника требует
   правки кода и редеплоя в 5+ местах.

5. ⬜ Декомпозиция ContractTabs.tsx — файл огромный (тысячи строк, десятки
   useState), каждая правка требует grep по всему repomix. Не спринт-фриз,
   а отдельная спланированная задача, когда не будет горящих багов:
   вынести вкладки в отдельные компоненты (ApprovalTab, ChatTab, DocumentsTab,
   ExecutionTab), состояние карточки — в React Context или Zustand.

6. ⬜ Race condition при генерации номера договора — между select максимума
   и insert другой пользователь может создать документ с тем же номером.
   Актуально при 50-60 одновременных пользователях (для этого и брали
   Vercel Pro). Решение: уникальный констрейнт на contracts.number + retry
   при конфликте, либо SELECT ... FOR UPDATE / отдельная таблица-счётчик.

7. ⬜ lib/contractStatus.ts — единый модуль меток/цветов СТАТУСА ДОГОВОРА
   (не путать со статусами участника согласования — тот словарь уже единый
   в ContractTabs.tsx). Статус договора (черновик/на_согласовании/
   согласован и т.д.) дублируется в нескольких компонентах списка/дашбордов.

8. ⬜ Проверить и убрать дефолтный JWT-секрет OnlyOffice, если в коде
   есть fallback вида `?? 'какой-то-секрет-по-умолчанию'` — секрет должен
   браться строго из ENV, при отсутствии — явная ошибка, не тихий дефолт.

9. ⬜ postMessage как альтернатива/дополнение к refresh_id для ОСТАВШЕЙСЯ
   нерешённой части deep-link проблемы — прямой клик по ссылке ВНУТРИ
   iframe чата Б24 (см. раздел 7). refresh_id решил только сценарий
   отдельной вкладки. Общение iframe ↔ родительское окно Б24 через
   window.postMessage — стоит проверить, поддерживает ли это API Битрикс24
   для безопасного получения контекста без потери contract_id.

10. ⬜ «Паспорт контрагента» / risk-паттерны по истории договоров —
    продуктовая идея на будущее, не спринт-задача. Например: «с этим
    контрагентом уже 11 договоров, во всех была предоплата, в новом её
    убрали — это отклонение от паттерна». Вся инфраструктура для этого уже
    есть (контрагенты + история договоров + проверки надёжности в одной
    базе), не требует полного RAG/Knowledge Graph, можно реализовать точечно.

Также отмечено (согласны, но НЕ приоритет на ближайшие спринты — горизонт
6-12 месяцев, инженерная зрелость команды/продукта, не точечная задача):
крупный рефакторинг на Domain/Service Layer/Event Bus, интеграция с
ЭДО-операторами (Диадок/СБИС) для автостатуса подписания, вынос парсинга
PDF в отдельный микросервис, rate limiting, CSP/helmet, серверная проверка
identity вместо доверия bitrix_user_id из тела запроса (реальный риск —
не случайный взломщик, а нечестный сотрудник, подделывающий след в истории
действий; не пожарный приоритет для внутреннего инструмента на 223 человека,
но держать в уме при следующем крупном рефакторинге авторизации).
