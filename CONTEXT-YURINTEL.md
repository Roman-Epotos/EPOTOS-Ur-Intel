  деплой, хостит RAG-ассистента, используется через простой fetch

⚠️ КРИТИЧЕСКИ ВАЖНО: OpenRouter с 27.06.2026 блокирует API-запросы от
  аккаунтов из российского региона. ЮрИнтел НЕ затронут — запросы идут
  через Vercel (нероссийский IP). Похоже на IP-based блокировку, не
  account-based. Критично учесть при возможном переносе на Selectel
  (см. раздел 12) — перенос СЕРВЕРНОЙ части сломает ИИ-функционал без
  отдельного нероссийского маршрута для вызовов к OpenRouter.

=============================================================
Нумерация: ПРЕФИКС-ТИПКОД-ГОД/МЕСЯЦ/НН

ВАЖНО: фильтр deleted_at=is.null УБРАН из запроса генерации номера —
  мягко удалённый документ учитывается в максимуме, номер не переиспользуется
⚠️ ОСТАТОЧНЫЙ РИСК: race condition между select максимума и insert —
  см. раздел 12, пункт 6
ВАЖНО: Э-К — двойной префикс → company_prefix из БД или startsWith('Э-К')
НОВОЕ (08.07.2026): номер теперь можно менять вручную после создания —
  см. раздел 6 «Изменение номера документа». Уникальность номера
  проверяется при каждом изменении, формат — нет (гибкость для
  особых/legacy случаев).

Статусы: черновик → на_согласовании → согласован → [на_подписи_в_эдо →]
  загружен_частично → подписан → на_исполнении (↘ отклонён)
LOCKED_STATUSES = ['согласован','на_подписи_в_эдо','загружен_частично','подписан','на_исполнении']
⚠️ В ContractTabs.tsx есть ВТОРОЙ, отдельно захардкоженный похожий список
  для кнопки «+ Добавить» вложение (без 'на_подписи_в_эдо') — известный
  технический долг, не унифицирован с LOCKED_STATUSES, см. раздел 12.

Приём на хранение (storage_confirmed_*) НЕ является статусом — независимый
  бейдж поверх статуса, доступен когда status IN ('подписан','на_исполнении')

Баг авто-закрытия согласования (POST/DELETE participants):
  POST (добавление участника) — ПОСЛЕ реактивации завершённой сессии (если
    role !== 'observer') НЕ должен пересчитывать «все ли обязательные
    согласовали» — это может закрыть сессию обратно в ТОМ ЖЕ запросе,
    если добавляли необязательного участника (сам он не входит в проверку
    по required, а старые required уже были approved). Пересчёт полностью
    убран из POST.
  DELETE (удаление участника) — ЗАКОННО пересчитывает «все ли ОСТАВШИЕСЯ
    обязательные согласовали» после удаления — если да, закрывает сессию
    (contract→'согласован', session→'completed'). Это ЛЕГИТИМНОЕ поведение,
    специфичное для DELETE, НЕ должно копироваться в другие обработчики.
